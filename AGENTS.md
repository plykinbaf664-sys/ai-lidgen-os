# Leadgen OS — правила разработки

## Цель проекта

Собрать внутреннюю AI-first систему лидогенерации для собственного оффера.

На текущем этапе:

* ICP фиксирован и зашит в систему;
* оффер фиксирован;
* система не является универсальным SaaS-продуктом;
* не нужно проектировать избыточную универсальность без отдельного запроса.

Главная задача системы:

```text
Найти компании
→ найти лучший доступный вход
→ определить повод для контакта
→ создать персонализированное сообщение
→ подготовить follow-up
→ сохранить результат
→ передать в работу
```

---

# Telegram-first подход

Система проектируется как Telegram-first.

Пользователь не должен постоянно заходить на сайт.

Telegram является основным операционным интерфейсом.

Сайт является панелью управления.

Любое действие, которое можно выполнить через Telegram без потери качества управления, должно быть доступно через Telegram.

---

# Архитектура системы

Система состоит из двух контуров.

## 1. Web Panel

Используется для:

* запуска кампаний;
* настройки системы;
* просмотра лидов;
* просмотра статистики;
* просмотра аналитики;
* контроля работы системы.

## 2. Telegram Bot

Используется для ежедневной работы.

Система должна уметь:

```text
Найти лида
→ отправить карточку в Telegram
→ показать причину попадания в выборку
→ показать сигнал
→ показать hook
→ показать сообщение
```

Через Telegram пользователь должен иметь возможность:

```text
Одобрить лид
Отклонить лид
Поставить на паузу
Посмотреть детали
Запросить новую выборку
Посмотреть статус кампании
```

Главный принцип:

```text
Telegram = операционный центр

Web = центр управления и аналитики
```

---

# Главный принцип работы

Не пытаться собрать всю систему сразу.

Любая задача разбивается на маленькие законченные этапы.

Перед началом работы:

1. Составить короткий план.
2. Перечислить файлы, которые будут созданы или изменены.
3. Объяснить назначение каждого файла.
4. Только после этого приступать к реализации.

После завершения:

1. Показать список изменённых файлов.
2. Объяснить результат.
3. Дать инструкцию по проверке.
4. Предложить commit message.

---

# Архитектурный подход

Использовать разделение ответственности.

```text
app/
→ страницы и API

components/
→ интерфейс

lib/
→ бизнес-логика

agents/
→ агенты системы

supabase/
→ хранение данных
```

Не смешивать:

* UI;
* бизнес-логику;
* интеграции;
* API;
* хранение данных.

---

# Логика системы

Система строится как pipeline.

```text
Campaign
→ Finder
→ Contact Finder
→ Signal Detection
→ Personalizer
→ Message Writer
→ CRM
→ Telegram
```

Каждый этап должен быть отдельным модулем.

Запрещено писать огромные файлы с несколькими ответственностями.

---

# Контакты

Главная задача системы:

Не обязательно найти ЛПР.

Главная задача:

```text
Найти лучший доступный вход в компанию.
```

Приоритет:

```text
ЛПР
→ руководитель отдела
→ founder / owner
→ общий email
→ форма сайта
→ LinkedIn
→ соцсети компании
```

Если данные отсутствуют:

* не выдумывать контакты;
* не выдумывать email;
* не выдумывать людей;
* не подставлять фиктивные данные.

Использовать null.

Лучше честно вернуть отсутствие данных, чем придумать их.

---

# Сигналы

Любое сообщение должно строиться на реальном сигнале.

Примеры:

* найм сотрудников;
* активный запуск продукта;
* активная реклама;
* высокий контентный темп;
* рост компании.

Запрещено придумывать сигналы без подтверждения.

---

# Персонализация

Каждое сообщение должно строиться по схеме:

```text
Сигнал
→ Hook
→ Сообщение
```

Нельзя:

* писать шаблонный спам;
* использовать общие фразы;
* придумывать факты;
* делать персонализацию без данных.

---

# Автоматизация

На первом этапе система автоматически:

```text
ищет компании
ищет контакты
определяет сигналы
создаёт hook
создаёт сообщения
создаёт follow-up
сохраняет данные
отправляет карточки в Telegram
```

На первом этапе система НЕ занимается массовой автоматической отправкой сообщений.

Сначала строится качественный pipeline.

Автоматическая отправка подключается отдельным этапом.

---

# Интеграции

Планируемые интеграции:

```text
Supabase
OpenAI
Clay
Telegram Bot
```

Все ключи только через environment variables.

Никогда не хранить ключи в коде.

---

# Запреты

Без отдельного согласования запрещено:

* менять package.json;
* устанавливать новые библиотеки;
* менять архитектуру проекта;
* трогать .env;
* массово отправлять сообщения;
* переписывать работающие модули;
* добавлять сложные интеграции;
* ломать существующую структуру проекта.

---

# Что нельзя делать

Запрещено:

* собирать всю систему одним запросом;
* делать универсальный SaaS;
* усложнять архитектуру без необходимости;
* писать код "на будущее" без реального применения;
* создавать лишние сущности;
* создавать лишние абстракции;
* выдумывать данные;
* скрыто менять существующий код.

---

# Приоритет разработки

Всегда двигаться в следующем порядке:

```text
1. Рабочий pipeline
2. Telegram-интерфейс
3. Хранение данных
4. Интеграции
5. Автоматическая отправка
6. Аналитика
7. Масштабирование
```

---

# Критерий качества

Качество архитектуры важнее скорости.

Каждый новый модуль должен:

* решать конкретную задачу;
* быть изолированным;
* легко тестироваться;
* не ломать существующую систему;
* иметь понятную ответственность.

Главная цель:

```text
Построить надёжную систему лидогенерации,
которой можно пользоваться каждый день через Telegram,
а не набор красивых, но бесполезных функций.
```
